langChain基础

①在venv环境中安装langchain
pip install langchain
pip install langchain-community
pip install langchain-text-splitters
pip install beautifulsoup4 专门用来解析 HTML / XML 的 Python 库

②切片代码

关于separators参数解释
按优先级从高到低 的「切分符」。切分时会 尽量先在更靠前的分隔符上断开
1 \n\n 空行 / 段落：优先在段落之间断开，大块语义最完整。
2 \n 单行换行：段落内再拆成行。
3 。 中文（日文等）句号：按句切。
4 . 英文句号（小数点也可能被误切，视文本而定）。
5 “ “ 空格：常见于英文单词之间；中文一般没有空格，这一步对中文帮助小，但作为最后手段仍可把超长串硬拆小。

切分的思路

1. 按文档结构切，而不是只按长度

  • Markdown / HTML：尽量按标题（#、##）、

    等切块，再在块内用 RecursiveCharacterTextSplitter 控制上限。

  • PDF：若有版面信息，按页/段优于纯字符硬切。

  • 原因：一块里主题越单一，向量语义越集中，和用户问题越容易「对上」。

LangChain 里可考虑 MarkdownHeaderTextSplitter、HTMLHeaderTextSplitter 等与递归切分组合。

2. separators 贴合你的语言与排版

  • 中文为主：把 。、！、？ 等放进列表，顺序在 . 之前或并列。

  • 技术文档：可加 \n- 、\n1.（列表）等，避免把一条列表拆成两半。

  • 先 段落 \n\n 再 句末，再 空格，和之前说的一样：先语义大块，再细切。

3. 用 chunk_overlap，减少「关键句卡在边界」

  • 问答句刚好跨两个 chunk 时，没有 overlap 容易 两边都搜不到或只搜到半句。

  • 常见起点：overlap ≈ chunk 的 10%～20%，再按召回结果微调。

4. 控制的是「语义单位」，优先用 token 意识（长文本时）

  • 默认按 字符 的 chunk_size，和 embedding 的 token 上限 不是一回事。

  • 中文密集文本：用 tokenizer 抽几条 chunk 看 token 数，避免顶满模型上限或浪费上下文。

  • 需要时改用 按 token 切 的 splitter，和所用 embedding 模型一致。

5. 元数据（metadata）比「切得更碎」更能提升「精确感」

切片时给每条 chunk 带上 source（URL/文件名）、title、section、page 等，检索后可以：

  • 按元数据过滤（只搜某文档、某章节）；

  • 把标题拼进 chunk 正文前一行（「假标题」），向量里带上章节语义。

LangChain 的 Document 支持 metadata，很多 loader 已带部分字段。

6. 小 chunk 检索 + 大 chunk 喂给 LLM（parent document）

  • 索引用小 chunk（向量更准、定位更细）。

  • 命中后取 父级大段或整节 再送给 LLM，避免上下文不够。

LangChain 里有 Parent Document Retriever 一类模式；思路是「检索细、阅读粗」。

7. 清洗再切：减少噪声向量

  • 导航栏、页脚、版权声明、重复模板，切进去会变成 高频但无意义 的块，冲淡检索。

  • WebBaseLoader 可先 BeautifulSoup 去 nav/footer，或只取 main 区域再 split。

8. 用实验闭环，而不是一次定死参数

  • 固定一批 真实用户问题，看：Top-k 里是否总出现「该在的段落」。

  • 调整 chunk_size / overlap / 分隔符 / 是否按标题切，对比命中率再定。 —一句话： 想更精确，优先 按标题/段落结构切 + 合理 overlap + 丰富 metadata（必要时父文档），其次才是微调 chunk_size；干净正文比盲目切更碎更重要。**

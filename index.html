④调用LLM
pip install google-genai 安装

代码：
from google import genai
from dotenv import load_dotenv

load_dotenv()
genaiClient = genai.Client()
answer = generate_answer(question, reranked_docs, genaiClient)

def generate_answer(question: str, answer_docs: list, client: genai.Client)->str:
    prompt = f"""
    あなたはヘルプデスクのアシスタントです。以下の質問に対して、適切な回答を生成してください。
    質問に対して、以下のドキュメントを参考：
    {"\n".join(answer_docs)}

    質問: {question}
    """
    print(prompt)

    response = client.models.generate_content(
        model="gemini-2.5-flash",
        contents=prompt,
    )
    return response.text

langChain基础-embedding

①安装sentence-transformers
pip install sentence-transformers
误区：即使使用langChain接口调用embedding模型，也必须安装sentence-transformers
因为langChain接口只是对sentence-transformers进行了封装，底层还是需要

②调用BAAI/bge-m3模型，生成embedding

③安装chromadb
python3 -m pip install chromadb

④存入chromadb

get_or_create_collection函数的参数说明：

name：
集合（collection）的名字，字符串，你自己定，相当于 「这张向量表 / 这个库」的标识。
同一 client 里 名字唯一：没有就 创建，已有就 取出来用（get_or_create）。
后面 collection.add / query 都是往这个集合里读写。

metadata:
这是 建集合时带的元数据，用来影响 向量索引怎么算「远近」。
Chroma 默认用 HNSW 做近似最近邻检索。
hnsw:space 表示 距离/相似度空间：
“cosine”：按 余弦距离（与 余弦相似度 对应；向量已 单位化 时，和用内积排序常一致）。
还有常见取值如 “l2”（欧氏距离）、”ip”（内积 / inner product）。
为什么写 cosine：
你对 embedding 做了 normalize_embeddings=True 时，向量在单位球面上，用 cosine 很常见，语义是「方向有多接近」。
注意：
这类 metadata 一般要在集合「第一次创建」时定好；已有集合再改可能受限或需建新集合，以 Chroma 版本文档为准。
若没传，会用 默认距离（不同版本默认可能是 L2，以文档为准）。

langChain基础-分片

①在venv环境中安装langchain
pip install langchain
pip install langchain-community
pip install langchain-text-splitters
pip install beautifulsoup4 专门用来解析 HTML / XML 的 Python 库

②切片代码

关于separators参数解释
按优先级从高到低 的「切分符」。切分时会 尽量先在更靠前的分隔符上断开
1 \n\n 空行 / 段落：优先在段落之间断开，大块语义最完整。
2 \n 单行换行：段落内再拆成行。
3 。 中文（日文等）句号：按句切。
4 . 英文句号（小数点也可能被误切，视文本而定）。
5 “ “ 空格：常见于英文单词之间；中文一般没有空格，这一步对中文帮助小，但作为最后手段仍可把超长串硬拆小。

②切分的思路

1. 按文档结构切，而不是只按长度

  • Markdown / HTML：尽量按标题（#、##）、 等切块，再在块内用 RecursiveCharacterTextSplitter 控制上限。
  • PDF：若有版面信息，按页/段优于纯字符硬切。
  • 原因：一块里主题越单一，向量语义越集中，和用户问题越容易「对上」。

LangChain 里可考虑 MarkdownHeaderTextSplitter、HTMLHeaderTextSplitter 等与递归切分组合。

2. separators 贴合你的语言与排版

  • 中文为主：把 。、！、？ 等放进列表，顺序在 . 之前或并列。
  • 技术文档：可加 \n- 、\n1.（列表）等，避免把一条列表拆成两半。
  • 先 段落 \n\n 再 句末，再 空格，和之前说的一样：先语义大块，再细切。

3. 用 chunk_overlap，减少「关键句卡在边界」

  • 问答句刚好跨两个 chunk 时，没有 overlap 容易 两边都搜不到或只搜到半句。
  • 常见起点：overlap ≈ chunk 的 10%～20%，再按召回结果微调。

4. 控制的是「语义单位」，优先用 token 意识（长文本时）

  • 默认按 字符 的 chunk_size，和 embedding 的 token 上限 不是一回事。
  • 中文密集文本：用 tokenizer 抽几条 chunk 看 token 数，避免顶满模型上限或浪费上下文。
  • 需要时改用 按 token 切 的 splitter，和所用 embedding 模型一致。

5. 元数据（metadata）比「切得更碎」更能提升「精确感」

切片时给每条 chunk 带上 source（URL/文件名）、title、section、page 等，检索后可以：

  • 按元数据过滤（只搜某文档、某章节）；
  • 把标题拼进 chunk 正文前一行（「假标题」），向量里带上章节语义。

LangChain 的 Document 支持 metadata，很多 loader 已带部分字段。

6. 小 chunk 检索 + 大 chunk 喂给 LLM（parent document）

  • 索引用小 chunk（向量更准、定位更细）。
  • 命中后取 父级大段或整节 再送给 LLM，避免上下文不够。

LangChain 里有 Parent Document Retriever 一类模式；思路是「检索细、阅读粗」。

7. 清洗再切：减少噪声向量

  • 导航栏、页脚、版权声明、重复模板，切进去会变成 高频但无意义 的块，冲淡检索。
  • WebBaseLoader 可先 BeautifulSoup 去 nav/footer，或只取 main 区域再 split。

8. 用实验闭环，而不是一次定死参数

  • 固定一批 真实用户问题，看：Top-k 里是否总出现「该在的段落」。
  • 调整 chunk_size / overlap / 分隔符 / 是否按标题切，对比命中率再定。 —一句话： 想更精确，优先 按标题/段落结构切 + 合理 overlap + 丰富 metadata（必要时父文档），其次才是微调 chunk_size；干净正文比盲目切更碎更重要。**

③langchain的TextSplitter类型介绍

1. 按字符 / 递归（最常用）

类型 作用简述
RecursiveCharacterTextSplitter 按一组 separators 递归切，尽量在段落、换行、句号、空格等处断开，兼顾 chunk_size / chunk_overlap。RAG 默认首选。
CharacterTextSplitter 按单个分隔符简单切，不够「聪明」，大块仍可能切得很硬。

2. 按 Token（对齐模型上限）

类型 作用简述
TokenTextSplitter 用 tiktoken 等按 token 切，适合和 OpenAI 等模型的上下文上限对齐。
SentenceTransformersTokenTextSplitter 按 Sentence-Transformers 所用模型的 tokenizer 切，和该类 embedding 长度一致。

3. 按文档结构（标题、标记）

类型 作用简述
MarkdownHeaderTextSplitter 按 Markdown 标题层级（#、##…）切块，块里可带上标题路径作上下文。
HTMLHeaderTextSplitter 按 HTML 标题标签（如 h1–h6）切。
LatexTextSplitter 面向 LaTeX 结构的切分。

常与 RecursiveCharacterTextSplitter 组合：先结构切块，再控制每块最大长度。

4. 代码

类型 作用简述
Language / RecursiveCharacterTextSplitter 的 language 参数 对 Python、JS、Go 等用适合该语言的 separators（函数、类等边界）。
（社区还有）CodeTextSplitter 等变体，按语言选分隔策略。

5. 结构化数据

类型 作用简述
RecursiveJsonSplitter / JSON 相关 splitter 把 JSON 按层级拆成多个小 Document，避免整坨 JSON 当一块。

6. 语义切分（边界更「懂内容」）

类型 作用简述
SemanticChunker（等在 experimental 或文档中单独说明） 用 embedding 相似度 找「语义转折」再切，块更连贯，但成本高、速度慢。

具体类名和导入路径以当前安装的 langchain-text-splitters 版本文档为准。

7. 传统 NLP 分句 / 分词

类型 作用简述
NLTKTextSplitter 依赖 NLTK 做句子级切分。
SpacyTextSplitter 依赖 spaCy。
其他语言可能有专门 splitter（如韩文等）。

需要额外装 NLTK/spaCy 模型，部署要算依赖成本。

选型一句话

  • 通用 RAG 网页/纯文本：RecursiveCharacterTextSplitter。

  • 要严格卡 token：TokenTextSplitter 或与 embedding 配套的 token splitter。

  • Markdown/HTML 文档：HeaderTextSplitter + 递归二次切。

  • JSON API 结果：RecursiveJsonSplitter 一类。

  • 追求语义块且能接受算力：SemanticChunker。
